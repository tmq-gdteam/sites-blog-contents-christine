If you're evaluating "agentic" automation for the enterprise, here's a practical due-diligence list:

1. Ask how it observes software. Pixel/coordinate-based bots are brittle and break on every UI change. Agents that read the underlying semantic structure understand intent and self-correct. This is the difference between constant maintenance and resilience.

2. Demand auditability. The black box is the real blocker. Can you trace *why* the agent took an action? Insist on logged, tamper-evident decision trails before anything touches live systems.

3. Look for "dry run" verification. The safest deployments test against real systems without committing changes — validating before a single real dollar moves.

4. Pin down the resource footprint. A monitoring agent on every endpoint raises real IT fears about slowdowns. Make the vendor show the architecture, not just assert efficiency.

5. Verify the numbers and the governance. Treat accuracy and savings metrics as claims to confirm. Confirm data isn't used to train public models, and that deployment can live inside your own infrastructure.

6. Reward honesty about limits. A vendor who acknowledges the edge cases where a human stays in the loop is more credible than one promising flawless autonomy.
